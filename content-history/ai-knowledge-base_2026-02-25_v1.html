I Built a Knowledge Base That My AI Can Actually Use

How a folder of markdown files became the most productive tool in my workflow

I've been using Cursor with Claude as my primary working environment for a few months now. Not just for code. For product strategy, marketing copy, research, decision-making, and content generation.

Somewhere along the way I realized the bottleneck wasn't the AI. It was me. I kept starting fresh conversations, re-explaining context, watching the model reach the same conclusions I'd already arrived at last week. Every new chat was amnesia. Good amnesia, because the model is smart. But amnesia.

So I started building a knowledge base. Not a wiki. Not a Notion workspace. A folder of markdown files in a git repo, organized by topic, written in plain language, and designed so that an AI agent could get up to speed on my entire project in under 30 seconds.

It changed everything about how I work.

The Setup

The repo is simple. A root-level README that acts as a table of contents. Subdirectories for each domain: vision, products, technical architecture, research, marketing, decisions. Each file covers one topic. Each file starts with a one-line summary.

No database. No build step. No dependencies. Just .md files in folders, tracked by git.

The README is the most important file. When an agent opens a new conversation and I point it at the repo, the first thing it reads is the README. From that single file, it knows what exists, where to find it, and what each document covers. It can then read only the files it needs for the current task.

How the Agent Actually Navigates It

This is the part that surprised me.

I assumed the agent would follow hyperlinks between documents, the way a human reads a wiki. Click a link, read that page, click another link, follow a trail. That's how I'd navigate it.

The agent doesn't do that at all. It searches.

When it needs information, it runs parallel keyword searches across every file in the repo simultaneously. If I ask about a specific concept, it greps for that term across all 30+ files and gets back a list of every file and line where it appears. Then it reads the relevant files in full, in parallel. Two tool calls and it has complete context.

This means the agent doesn't care about the link structure. It cares about two things: the README (so it knows what exists) and consistent terminology (so its searches return clean results).

That second point took me a while to understand. If I call the same concept "pattern matching" in one document, "regex layer" in another, and "deterministic parsing" in a third, the agent has to run three separate searches to find all references. If I always call it "regex layer," one search finds everything.

I started thinking of this as registered terminology. Every concept in the project gets one canonical name. I use that name everywhere. The agent's search becomes a lookup instead of a scavenger hunt.

What It Replaced

Before the knowledge base, my workflow looked like this:

  • Start a new conversation
  • Spend 5–10 minutes explaining context
  • Get useful output
  • Lose all of that context when the conversation ends
  • Repeat

The knowledge base collapsed steps 1 and 2. Now I start a conversation, point it at the repo, and say "get up to speed." The agent reads the README, pulls the files it needs, and is ready to work in seconds. All the context I used to re-explain lives in the documents.

But the bigger change was less obvious. The knowledge base became the canonical record of every decision, every research finding, every piece of strategy. Before, that stuff lived in chat transcripts that nobody would ever re-read. Now it lives in files that are searchable, diffable, and version-controlled.

When I make a decision, I write it down in the decisions log with the date and the reasoning. When the agent starts a new conversation weeks later and I ask about that topic, it finds the decision. It doesn't re-derive it from first principles. It reads what I already decided and builds from there.

Writing for an AI Reader

I changed how I write documents once I understood how the agent reads them.

Short files, not long ones. If a single file grows past 500 lines, the agent has to search within it or read it in chunks, which means it might miss context. Splitting a long document into focused files means each file gets read in full. The agent gets complete context for the topic without wading through unrelated material.

Front-matter summaries. Every file starts with a one-line description, usually as a blockquote. The agent can read the first few lines of a file and decide whether it needs the rest. This saves time on large repos.

One topic per file. If a product spec also contains research notes and a marketing draft, searches return that file for too many queries. Noise goes up, relevance goes down. Keep files focused.

Descriptive filenames. nlu-intent-research.md tells the agent what's inside without opening it. notes-v2-final.md tells it nothing.

What It's Good At Now

With 30+ documents covering the full project, the knowledge base handles a few things surprisingly well.

Starting new conversations cold. I don't prep the agent anymore. I just say "read the repo and get up to speed" and it does. Thirty seconds later it can answer detailed questions about architecture, marketing strategy, competitive landscape, or product specs.

Generating prompts for other agents. When I need to spin up a separate agent for a specific task (building a demo, writing code, designing a feature), I can ask the knowledge base agent to generate a detailed prompt that includes all the relevant context from the repo. The second agent starts with full context without me having to assemble it manually.

Maintaining consistency across content. Blog posts, event descriptions, slide decks, product copy. All of it draws from the same source documents. When I update the product vision, every piece of content generated after that update reflects the new direction. No drift between what the website says, what the blog says, and what the slide deck says.

Catching contradictions. If I write something in a new document that conflicts with an earlier decision, the agent notices. It's searched the full repo; it knows what I said before. This is the closest thing I've found to an AI with institutional memory.

What I'd Do Differently

I started the knowledge base after months of conversations. A lot of context had to be reconstructed from chat transcripts. If I were starting a new project today, I'd create the repo on day one and write the first decision log entry before writing any code.

I'd also establish the terminology glossary early. The inconsistency problem compounds. Once you've written 20 documents with three different names for the same concept, normalizing them is tedious. Pick your terms upfront and stick with them.

And I'd keep the README ruthlessly maintained. It's the agent's entrypoint. If a file isn't in the README, the agent might not know it exists. If an obsolete file is still listed, the agent wastes time reading something that's no longer relevant.

Why Markdown, Why Git

People ask why I don't use Notion or Confluence or some purpose-built tool.

Markdown is plain text. Every AI model can read it natively. No API integration, no export step, no formatting translation. The agent reads the file directly.

Git gives me version history for free. I can see when a decision changed, what the previous version said, and who (or what) wrote it. When the agent drafts content, I review the diff before committing. The repo is the collaboration layer between me and the AI.

And the whole thing is portable. It's a folder. I can move it, copy it, back it up, open it in any editor. There's no vendor lock-in because there's no vendor.

The Takeaway

The best thing I did for my AI workflow wasn't finding a better model or writing better prompts. It was giving the model something to remember.

A knowledge base built for AI consumption turns every new conversation from a cold start into a warm handoff. The agent reads, searches, and picks up where the last conversation left off.

If you're doing serious work with AI and you're still relying on chat history to carry context, try this instead: make a folder, write some markdown files, keep a README, and point your agent at it. You'll be surprised how much faster everything gets when the AI can read your mind on the first message.
